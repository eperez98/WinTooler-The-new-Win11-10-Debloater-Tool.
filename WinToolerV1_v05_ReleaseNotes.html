GUI Boot Sequence

Every launch prints a live status table to the CLI window before the GUI opens. New steps introduced in v0.5 are highlighted.

#StageDescription
01 STA Thread Check Detects MTA mode — auto-restarts with -STA if needed (WPF requirement)
02 Root Detection 3-tier path fallback — works from bat file, PowerShell direct, or right-click
03 Admin Verification Confirms Administrator rights — exits cleanly if not elevated
04 ⭐ Restore Point Creates system restore point before touching anything. Bypasses 24hr Windows cooldown automatically via registry key
05 winget Bootstrap Locates winget. If absent: fetches VCLibs + UI.Xaml prereqs, downloads latest .msixbundle from GitHub API
06 ⭐ NuGet Provider Downloads DLL directly from Microsoft CDN — no interactive yes/no prompt (was broken in previous builds)
07 PSWindowsUpdate Installs from PSGallery if absent — needed for Windows Update tab
08 Execution Policy Relaxes to RemoteSigned if set to Restricted or AllSigned
09 .NET Framework Reads registry release key — detects 4.5 through 4.8.1 via if/elseif chain (fixed from broken switch)
10 winget Sources Runs winget source update to refresh package metadata before any installs
11 ⭐ App Update Scan Runs winget upgrade, parses all available updates, stores in memory — shown in App Updates tab with count badge
12 Module Load Dot-sources repair.ps1, tweaks.ps1, updates.ps1
13 Config Load Loads apps.json (111), tweaks.json (23), services.json (18)
14 ⭐ Startup Screen Language + Theme picker opens. Selection applies to entire main GUI instantly

Bug Fixes

  • F1
    GUI instant-close on launch — $PSScriptRoot was empty when launched via Launch.bat, so the script couldn't locate gui.ps1. Fixed with a 3-tier path fallback: $PSScriptRoot → $MyInvocation.MyCommand.Path → $PWD.Path.
  • F2
    WPF window killed instantly (MTA mode) — WPF requires STA threading. PowerShell 5.1 defaults to MTA. Added -STA to Launch.bat and a self-restart block that detects non-STA and relaunches automatically.
  • F3
    XAML XML parse crash — Unicode box-drawing characters were converted to --- during editing, making XAML comments like <!-- --- --> illegal XML (double-dash in comments). All such patterns cleaned; window title sanitised.
  • F4
    LetterSpacing crash on .NET Framework — LetterSpacing is a .NET 6+ WPF property. PowerShell 5.1 uses the older .NET Framework WPF which doesn't support it. Attribute removed.
  • F5
    Install tab showed no apps — Two root causes: (1) $script: scope inside dot-sourced functions pointed to the wrong script context — fixed by switching shared variables to $global:. (2) Set-Variable -Name "script:AppCatalog" created a literal variable named script:AppCatalog instead of a scoped one — fixed with direct assignment.
  • F6
    Services tab crash — $action: scope error — PowerShell interprets "Service $action: $svcName" as a scope-qualified variable reference ($action: like $env:). Fixed by delimiting as ${action}.
  • F7
    .NET version showing all matches — switch ($true) falls through all matching conditions in PowerShell unless break is used. The version string printed as "4.8.1 4.8 4.7.2 4.6.2". Replaced with an if/elseif chain.
  • F8
    NuGet provider interactive prompt ignored — Install-PackageProvider always prompts interactively even with -Force. The user's N keypress was consumed but the install ran anyway via a race condition. Fixed by downloading the DLL directly from cdn.oneget.org — zero prompts.
  • F9
    Restore point falsely reported as created — Windows silently skips restore point creation if one was made in the last 24hrs, but the catch block still printed "created". Now detects the cooldown, sets SystemRestorePointCreationFrequency=0 and retries, or reports the skip accurately.
  • F10
    SFC + DISM freezes the GUI — Both commands blocked the WPF dispatcher thread, making the window completely unresponsive for up to 15 minutes. Fixed with Start-Job async execution + Dispatcher.Invoke streaming of live output to the console box.
  • F11
    App updates reporting failures incorrectly — Exit code -1978335189 ("no applicable update") was shown as a failure. This happens when winget's metadata disagrees with the installer's version. Now auto-retried with --force; if it succeeds, counts as updated; if not, removed from list as "already current" rather than "failed". Exit code -1978335230 (hash mismatch) now clears the winget LocalCache and retries once.

Winget Exit Code Handling

Exit CodeMeaningHandled
0 Success ✓
-1978335138 Scope error (machine scope rejected) ✓ Auto-retry with --scope user
-1978335189 No applicable update found (version mismatch) ✓ Auto-retry with --force
-1978335230 Installer hash mismatch ✓ Cache cleared + retried once
-1978335212 Hash mismatch variant ⚠ Flagged — retry later
-1978335216 Package not found in any winget source — Reported as not found
-1978334972 Installer blocked by policy — Logged, reported

Known Issues / Limitations

  • ⚠ Heavy template tweaks are irreversible. Removing Microsoft Bloatware and Xbox components via the Heavy template cannot be undone with the Undo button. The boot-time restore point is your safety net.
  • ⚠ winget package availability varies. Some apps in the catalog may not be available in all regions or may be temporarily delisted from winget sources. These are reported as "Not found" with exit code -1978335216.
  • ⚠ PSWindowsUpdate module required for Updates tab. If PSGallery is blocked by your organization's network policy, the Windows Update tab will error. All other tabs remain fully functional.
  • ⚠ Language cannot be changed after launch without restarting the app. The startup screen only appears once per run. This will be addressed in a future release with an in-GUI language toggle.
  • ⚠ System Protection must be enabled on the C: drive for restore points to work. WinToolerV1 attempts to enable it automatically, but some managed environments may block this via Group Policy.

How to Install

  • Download WinToolerV1_v05_BETA.zip from the releases page
  • Extract anywhere — Desktop, Downloads, C:\Tools, etc.
  • Right-click Launch.bat → Run as administrator
  • Click Yes on the UAC prompt — watch the CLI boot sequence complete
  • On the startup screen: pick English or Español, then Dark or Light — click Launch WinToolerV1
No installation required. WinToolerV1 runs entirely from the extracted folder. No registry entries are created for the app itself, no background services are installed, and it can be deleted by simply removing the folder.
